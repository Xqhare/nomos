# Nomos Spec v0

Nomos is a human-readable and Markdown-compatible task management protocol.
It is heavily inspired by [todo.txt](https://github.com/todotxt/todo.txt) but incompatible with it.

It also takes inspiration from the [XDG Base Directory Specification](https://specifications.freedesktop.org/basedir/latest/) in terms of being as prescriptive as possible.

Nomos files (e.g., `TODO.md`) are 100% compliant with standard Markdown list rendering.
However, not all Markdown features are supported or made use of.

Put another way, a Nomos file is just a giant mixed Markdown list with some special formatting.

## Overview & Goals

I designed Nomos with the issue of cross-project issue and task management in mind.
I have also tried to make it as simple, extensible and generic as possible to support a wide range of use cases.

I have distilled my ideas down to these four design principles:

- **Data Sovereignty**: Tasks live directly with the code in markdown/text files.
- **Tool-Agnostic**: Read and write tasks in Neovim, standard Markdown previewers, or via custom CLI/TUI.
- **Zero External Dependencies**: No dependencies are needed.
- **Cross-Project Linkage**: Resolve dependencies and priority order across multiple projects using Kahn's topological sort.

## Notes for Implementers

This specification covers only the syntax and semantics of single Nomos files.
It is up to the individual implementers to decide the scope of tracked projects, how to track them and how permissive the parsing is.

To make is easier to follow, this repository also contains the reference implementation for a parser of Nomos files.

> The specification only makes use of three prescriptive adjectives: _should_, _must_ and _must not_.
> They are used and are to be interpreted as described in [RFC2119](https://datatracker.ietf.org/doc/html/rfc2119).

Implementers are free to impose arbitrary limits, restrictions and extensions on the syntax and semantics of Nomos files (such as a maximum nesting limit of tasks and notes) as long as they are well documented and clearly communicated.

The way implementations return values and errors is up to the implementer.
Any implementation may choose for example to reject an entire file if syntax rules are violated, or if a dependency can not be resolved.
It may also choose to return a partially parsed file with an attached list of errors.

### Scope of Tracked Projects

Nomos is best used across multiple projects.
It is up to the implementer to decide how to track them.

The reference implementation in this project for example utilises a passed in global configuration `.json` file containing a list of root directories for the parser to crawl through.
The exact behaviour is up to the implementer.

### Permissiveness of Parsing

The reference implementation rigidly enforces the specification to the letter, returning an error if it is violated in any way.

Markdown allows multi-line text for a single bullet point. The reference implementation does not.
Should any implementation decide to relax this rule, it should be done with care considering how the parser is going to distinguish a continued line from a note or a task.

Dangling Notes are another potential issue that _should_ be considered by the implementer. The reference implementation throws an error is a note can not be clearly associated with a specific task.

The reference implementation requires that a dependency to another task be declared by using the full unique title.
It is worth considering relaxing this rule to allow for partial matches, especially should the partial match be unambiguous.

## Syntax Specification

Nomos files are organized into tasks and notes using standard Markdown list rendering.

Following the syntax of [todo.txt](https://github.com/todotxt/todo.txt), tasks and notes are delimited by leading `-`, `*`, or `+` markers and following newlines.

### Grammar

Following the `todo.txt` specification, escaped characters are not supported.
Should an implementer decide to add support for them, it is recommended to use the `\` escape character for broad compatibility.

#### List Markers

To distinguish tasks from notes at a glance in plain-text, specific list markers are enforced:

- `-` (Hyphen): Declares a **Task**. _Must_ be followed by a status bracket (e.g., `- [ ]`).
- `*` (Asterisk): Declares a **Note** containing details, comments, or design thoughts.
- `+` (Plus): Reserved for future system extensions.

#### Notes

Notes are lines that start with `*`.

They may contain any characters, excluding newlines.

Notes _must_ always have an associated task.
Notes without an associated task are called danging notes, and special care must be taken in how an implementer handles them.

Notes may contain tags but not dependencies.

Tags are further explained [below](#tags).

#### Tasks

```text
- [Status] (Priority) Title :: {InceptionDate} {CompletionDate} Description +kindTag @locationTag keyTag=valueTag dep="Dependency Title"
```

Tasks always start with a `-` dash, followed by a space. Then, the status, priority, title, separated by the `::` delimiter from the dates and description.

- **Status** (Mandatory):
  - `[ ]` = Open
  - `[/]` = In Progress
  - `[x]` = Done
  - `[B]` = Blocked
  - `[D]` = Deferred
  - `[C]` = Cut / Dropped / Excluded
- **Priority** (Optional): A single uppercase letter, e.g., `(A)`.
- **Title** (Mandatory): The string preceding the `::` delimiter (excluding the status and priority brackets). Can contain spaces. _Must not_ contain `"` in any position.
- **Delimiter** (Mandatory): ` :: ` separates the core identity (title) from metadata and description. Please note the leading and trailing spaces.
- **Dates** (Optional): Immediate dates following `::` _must_ be in `YYYY-MM-DD` format.
  - One date: Inception.
  - Two dates: Inception & Completion.
- **Tags** (Optional):
  - `+kind`: Semantic categorization (e.g., `+bug`, `+feature`, `+release`).
  - `@location`: Specific file/subsystem/other (e.g., `@lib.rs`, `@executor`, `@homelab`).
  - `key=value`: Custom metadata (e.g., `est=2d`, `owner=xqhare`).
  - `#generic`: A generic tag for general notes (e.g., `#todo`, `#note`, `#question`, `#needsRethink`).
- **Dependencies** (Optional): Declared via `dep="Title"` (same project) or `dep=project_name:"Title"` (cross-project). Multiple dependencies are allowed.

##### Status

The status is mandatory. In contrast to `todo.txt`, where the status may only be either `[ ]` or `[x]`, Nomos allows any of the following:

- `[ ]` = Open
- `[x]` = Done
- `[/]` = In Progress
- `[B]` = Blocked
- `[C]` = Cut / Dropped / Excluded
- `[D]` = Deferred

This allows any Markdown viewer to correctly render open and done tasks, while extending the syntax to allow for other states.

##### Priority

The priority is optional.

It may range from `A` to `Z` and is case-insensitive.
It is recommended to use uppercase.

##### Title

A title may be of any length and contain spaces, but no newline characters.
It _must not_ contain `"` in any position.

Any title of a project _must_ be unique for that project.

##### Delimiter

The `::` delimiter separates the core identity (status, priority, and title) from metadata and description.

It _must_ be preceded and followed by a single space for readability.

##### Dates

The dates are optional.

They _must_ be in `YYYY-MM-DD` format.

The dates are immediately following the `::` delimiter.

- One date: Inception.
- Two dates: Inception & Completion.

##### Description

The description is entirely optional.

It may be of any length.

It _must not_ contain a newline character in any position.

The description is all text, besides the optional dates, after the `::` delimiter.
This includes any tags and dependencies.

###### Tags

Tags _must_ only be part of the description or used inside a note.

The `kind` and `location` tags function in the same way as `todo.txt`'s `context` and `project` tags.
They do not allow for any whitespace.

In contrast, the `key=value` tags do not function in the same way as `todo.txt`'s `special` tags.

The `generic` tag is unique to Nomos. It _should_ be used for any and all general text a user wants to be easily sorted by and searched for.

All tags allow for single word values as well as the POSIX `"VALUE WITH SPACES"` format (resulting in `key="VALUE WITH SPACES"` or `#"ANOTHER VALUE"`), but the value _must_ be wrapped in double quotes.

> [!note] 
> The symbols `+`, `@` and `#` _should_ only be parsed as a tag if they are followed by non-whitespace characters.
> If used on their own they _should_ be parsed as part of the text they are found in.

###### Dependency Rules

Dependencies are declared via `dep="Title"` (same project) or `dep=project_name:"Title"`.
Only the title itself may be and _should_ be wrapped in double quotes.

If a dependency is located in another project, `project_name` _must_ be specified.
Any project name _must_ be unique among all tracked projects.
The project name is case-sensitive.

Nomos assumes that all projects are git repositories and as such have no spaces in their names.
Adhering to, or breaking from, this assumption is up to the implementer but must be carefully taken into account.
Should whitespace characters be permitted, the project name _should_ be wrapped using the `"` character.

The parsing of the dependency _must_ be case-sensitive.

How to handle declared dependencies that do not exist is up to the implementer.
Any implementation _should_ at least warn the user that the declared dependency does not exist.

> [!important] 
> Dependencies are not allowed to be declared in notes.

> [!important] 
> Dependency cycles are _not_ allowed. (Task A depends on B, which depends on A.)
> While mutual depending tasks do happen in real life, Nomos does not support them for simplicity.

### Hierarchy & Subtask Dependencies
Hierarchy is established using indentation (4 spaces per level):
```markdown
- [ ] (A) Integrate CLI toolkit :: 2026-05-22 Integrate Eshu +feature @src/main.rs
    * Do not forget to update Cargo.toml
    - [ ] Setup argument builder :: Write command definitions
    - [x] Parse subcommands :: Test parser against standard inputs
        * Remember to check for std::env::args_os compatibility
        * Make sure we don't pull in any external parser dependencies
- [B] Run Kahn Sort :: dep="Integrate CLI toolkit" +feature @src/graph.rs
```

Here, the first task (`Integrate CLI toolkit`) is a parent task that has two child subtasks (`Setup argument builder` and `Parse subcommands`) as well as one associated note (`Do not forget to update Cargo.toml`).
The first child (`Setup argument builder`) has neither children nor notes.
The second child (`Parse subcommands`) has two notes (`Remember to check for std::env::args_os compatibility` and `Make sure we don't pull in any external parser dependencies`).
Lastly, the second task (`Run Kahn Sort`) is a task that has no subtasks or notes but depends on the first task to be completed.

**Subtask Dependency Rules**:
1. A parent task is implicitly dependent on all its child subtasks. A parent task cannot be marked completed (`[x]`) until all its child subtasks are completed.
2. In Kahn's topological sort, child subtasks are scheduled *before* their parent task.
3. Notes (`*`) are associated with the task directly above them. They are indented to the next level as the task they are associated with.
4. Child tasks are associated with the task above them that is one indentation level lower.
