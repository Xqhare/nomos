# Nomos Spec v1

Nomos is a human-readable and Markdown-compatible task management protocol. 
Version 1 (v1) builds upon the foundational v0 specification, relaxing strict parser requirements to allow embedding tasks within standard markdown files, transitioning to a `.nomos` file extension, introducing a numeric priority scale, and enabling unified subtask dependency scheduling.

It remains heavily inspired by [todo.txt](https://github.com/todotxt/todo.txt) but incompatible with it.

---

## Notes for Implementers

This specification defines the syntax and semantics of Nomos v1 files. Implementations must support v1 parsing rules alongside legacy v0 rules, resolving version contexts using the precedence resolution pipeline.

### Version Resolution Pipeline
When a Nomos engine processes a file, the parser resolves the format version using the following precedence order:

1. **In-File Metadata Override (Optional)**: If the first non-empty line of the file is an HTML comment matching `<!-- nomos: X -->` (where `X` is `0` or `1`), version `X` is used. This allows standalone files to be self-describing outside of any project context.
2. **Project Configuration (`nomos.json`)**: If a `version` key exists in the project configuration (e.g., `"version": 1`), all files in that project default to that version.
3. **Global Configuration (`config.json`)**: If a `version` key exists in the global configuration, it acts as the default fallback.
4. **Extension Inference**: If no version is explicitly configured or declared:
   * File extension `.md` is parsed using the **v0** rules.
   * File extension `.nomos` is parsed using the **v1** (or latest binary) rules.

## Syntax Specification

Nomos v1 files are written in standard Markdown list syntax but use relaxed formatting rules to allow coexistence with documentation prose.

### Grammar

#### List Markers
* `-` (Hyphen): Declares a **Task**. Must be followed by a status bracket (e.g., `- [ ]`).
* `*` (Asterisk): Declares a **Note** containing details, comments, or annotations.
* `+` (Plus): Reserved for future system extensions.

#### Notes
Notes are indented lines starting with `* `.
* Notes must be associated with a parent task directly above them.
* Notes may contain tags but must not declare dependencies.
* Under v1, any indented line that does not begin with the task marker `- ` or note marker `* ` is ignored by the parser. This prevents standard text blocks and prose paragraphs within markdown files from being parsed as tasks or notes.

#### Tasks
Tasks follow the general structural format:
```text
- [Status] (Priority) Title :: {InceptionDate} {CompletionDate} Description +kindTag @locationTag keyTag=valueTag dep="Dependency Title"
```

* **Status** (Mandatory): Represents the lifecycle of the task.
  * `[ ]` = Open
  * `[/]` = In Progress
  * `[x]` = Done
  * `[B]` = Blocked
  * `[D]` = Deferred
  * `[C]` = Cut / Dropped / Excluded
* **Priority** (Optional): A single digit `(0)` through `(9)`.
* **Title** (Mandatory): The string representing the identity of the task.
* **Delimiter** (Optional): ` :: ` separates the title from dates and description.
* **Dates** (Optional): Immediate dates following `::` must be in `YYYY-MM-DD` format.
* **Description** (Optional): All trailing text, including tags and dependencies.

#### 2. Numeric Priority Scale
Priority is represented by a single digit `0` to `9` instead of alphabetic letters:

| Number | Priority Level | v0 Equivalents |
| :---: | :--- | :---: |
| **0** | Extremely Important (Discouraged) | - |
| **1** | Most Important | A, B, C |
| **2** | Especially Important | D, E, F |
| **3** | Highly Important | G, H, I |
| **4** | Very Important | J, K, L |
| **5** | Important | M, N |
| **6** | Somewhat Important | O, P, Q |
| **7** | Slightly Important | R, S, T |
| **8** | Less Important | U, V, W |
| **9** | Least Important | X, Y, Z |

Implementers should treat tasks without an explicitly declared priority as having a default priority of `5` (Important) for scheduling purposes.

#### Title

A title may be of any length and contain spaces, but no newline characters.
It _must not_ contain `"` in any position.

Any title of a project _must_ be unique for that project.

##### Delimiter

The `::` delimiter separates the core identity (status, priority, and title) from metadata and description.

If present, it _must_ be preceded and followed by a single space for readability.

##### Dates

The dates are optional.

They _must_ be in `YYYY-MM-DD` format.

The dates are immediately following the `::` delimiter.

- One date: Inception.
- Two dates: Inception & Completion.

##### Description

The description is entirely optional.

It may be of any length.

It _must not_ contain a newline character in any position.

The description is all text, besides the optional dates, after the `::` delimiter.
This includes any tags and dependencies.


#### Tags

Tags _must_ only be part of the description or used inside a note.

There are four types of tags:

- `+kind`: Semantic categorization (e.g., `+bug`, `+feature`, `+release`).
- `@location`: Specific file/subsystem/other (e.g., `@lib.rs`, `@executor`, `@homelab`).
- `key=value`: Custom metadata (e.g., `est=2d`, `owner=xqhare`).
- `#generic`: A generic tag for general notes (e.g., `#todo`, `#note`, `#question`, `#needsRethink`).

The `kind` and `location` tags function in the same way as `todo.txt`'s `context` and `project` tags.
They do not allow for any whitespace.

In contrast, the `key=value` tags do not function in the same way as `todo.txt`'s `special` tags.

The `generic` tag is unique to Nomos. It _should_ be used for any and all general text a user wants to be easily sorted by and searched for.

All tags allow for single word values as well as the POSIX `"VALUE WITH SPACES"` format (resulting in `key="VALUE WITH SPACES"` or `#"ANOTHER VALUE"`), but the value _must_ be wrapped in double quotes.

> [!note] 
> The symbols `+`, `@` and `#` _should_ only be parsed as a tag if they are followed by non-whitespace characters.
> If used on their own they _should_ be parsed as part of the text they are found in.

#### Dependency Rules

Dependencies are declared via `dep="Title"` (same project) or `dep=project_name:"Title"`.
Only the title itself may be and _should_ be wrapped in double quotes.

If a dependency is located in another project, `project_name` _must_ be specified.
Any project name _must_ be unique among all tracked projects.
The project name is case-sensitive.

Nomos assumes that all projects are git repositories and as such have no spaces in their names.
Adhering to, or breaking from, this assumption is up to the implementer but must be carefully taken into account.
Should whitespace characters be permitted, the project name _should_ be wrapped using the `"` character.

The parsing of the dependency _must_ be case-sensitive.

How to handle declared dependencies that do not exist is up to the implementer.
Any implementation _should_ at least warn the user that the declared dependency does not exist.

> [!important] 
> Dependencies are not allowed to be declared in notes.

> [!important] 
> Dependency cycles are _not_ allowed. (Task A depends on B, which depends on A.)
> While mutual depending tasks do happen in real life, Nomos does not support them for simplicity.

## Key Differences from v0

### 1. Optional Title Delimiter
In v1, if a task only contains a title, the `::` delimiter is optional:
```markdown
- [ ] Simple Task Name
```
* If the delimiter `::` (or ` :: `) is omitted, the entire remaining string (after status and priority) is treated as the task title.
* In the absence of the delimiter, no dates, descriptions, tags, or dependencies can be parsed for that task line.

### 3. Unified Subtask Kahn Sorting
* Subtasks are treated as first-class nodes in Kahn's topological sort.
* **Nesting Rule**: Parent tasks depend on their child subtasks. A parent task is implicitly blocked until its subtasks are completed.
* **Sorting order**: In the generated scheduler queue, child subtasks are ordered *before* their parent task.
* **ID Unique Resolution**: Subtasks are identified in the topological sort graph via path-based unique IDs: `project_name:ParentTitle:SubtaskTitle`.

### 4. Done vs. Cut Status Validation
* **Explicit Dependencies**: A task marked as `Done` can only depend on tasks that are also `Done`. If an explicitly declared dependency (`dep=`) is marked as `Cut`, the dependent task is blocked from being marked `Done` (raising a validation error).
* **Implicit (Subtask) Dependencies**: A parent task can be marked `Done` if all of its subtasks are resolved (either `Done` or `Cut`). Unlike explicit dependencies, a parent task is allowed to be completed even if its subtasks were `Cut`.
* **Completion Date**: A completion date may only be set if the task and all of its subtasks are resolved (`Done` or `Cut`) and its inception date is set.
