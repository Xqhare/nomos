# Musings

## File Extension

The file extension for Nomos files is `.md`.
It may be advantageous to change this to `.nomos`. 

This would make crawling the filesystem and the LSP server easier.

### Open Question

Would it be mandatory to use the file extension `.nomos`?
Or would both be ok?

## Optional Title Delimiter

If a task only contains a title:

```markdown
- [ ] Task Name ::
```

would become

```markdown
- [ ] Task Name
```

Of course this means, that if a user forgets the delimiter between the title and description, and doesnt provide the optional dates, the entire thing would be taken as the title.

The only issue with that would be task linking, however the developer would quickly notice this and update the task accordingly.

## The Version Problem

In contrast to the xff spec, I do not have a way of tracking versions of files.

This is a problem for the following reasons:

- All projects are tracked with nomos v0 spec files at the moment.
- v1 will incorporate breaking changes from v0.

The largest issue flying in the face of most, if not all attempts to solve this is the Markdown compliance.

Defining that text must be added to the file (like the first line needing to be the version number) is stupid, and unergonomic to the extreme.

So, I dont want to change the contents of the file, and the bytestructure of Markdown cannot be changed eiter, making it impossible to track the verison of the file with the file itself.

### Optional Version KV Pair in Global & Project Config

I could add a `version` KV pair to the global config, and the project config.

This would allow the version to be tracked in a way that is compatible with Markdown.

If the key is not present, the latest nomos version is assumed.

In the global config this would mean that *ALL* projects would be parsed using the defined versions parser.

In the project config this would mean that only the project (but *ALL* tracked files in the project) would be parsed using the defined version parser.

### Automatic Version Update

Would require a way of saving entire files to disc.

This would need to be somewhat manual, probably as its own subcommand.

Something like `$ nomos update -v=0 -v=1` for example. This would parse *ONLY* v0 files, and replace them as valid v1 versions of themselves.

## Status and Priority

In v0 both status and priority are defined as a single character.
This can lead to some confusion while reading the file, and editing it.

Using A-Z for priority was inherited from todo.txt; and 26 different priorities is a bit too much.

Move prio to be a number, 0-9 OR 1-9.
Still single char, but more logical and easier to separate intuitively from status.

0 is a special case; Should it mean most important? 1 would be the best fit for that.
Moving it to least important just does not make sense logically (9 < 0 ??).
Removing it as a valid prio would fly in the face of developer expectations though.

Maybe best to define 0 as a valid, but discouraged, `Extreme` prio level.
