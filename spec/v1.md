# Musings

## Issues to keep in mind

### Done and Cut

Both `done` and `cut` statuses are sometimes the same thing.

In the case of a depending task, it may only be done if the task it depends on is done.
If it was cut, removing the dependency automatically can lead to misscomunication.
(I though you had fixed bug A, so I went ahead with feature C only to find out fixing bug A was cut)

In the case of a parent task however (where the dependence is not explicit), the parent should be able to be marked as done even if all sub tasks are marked as `cut`.

### Kahns

Currently dependencies of sub tasks are ignored.
Thats wrong.

Easy fix: push each sub task into the flattened kahn list with the parent depending on it. rest should work as is.

But this is work for the parser side, not file syntax.

It would turn into a specific way of writing a markdown list.

## Optional Title Delimiter

If a task only contains a title:

```markdown
- [ ] Task Name ::
```

would become

```markdown
- [ ] Task Name
```

Of course this means, that if a user forgets the delimiter between the title and description, and doesnt provide the optional dates, the entire thing would be taken as the title.

The only issue with that would be task linking, however the developer would quickly notice this and update the task accordingly.

ACK
This is a good idea. Makes writing small / simple tasks a lot easier.
To keep complexity low, everything encountered until the delimiter OR first line is treated as the title (tags, dates and all if missing).

## The Version Problem

In contrast to the XFF spec, Nomos files are standard Markdown text files, making it challenging to track format versions without breaking Markdown compatibility or introducing ugly boilerplate.

To solve this, v1 uses a **hybrid versioning model** that combines extension-based routing, project-level configuration, and optional in-file metadata overrides.

### Version Resolution Pipeline

When Nomos processes a file, it resolves the format version using the following precedence order:

1. **In-File Metadata Override (Optional):** If the first non-empty line of the file is an HTML comment matching `<!-- nomos: X -->`, version `X` is used. This allows standalone files to be fully self-describing outside of any project context.
2. **Project Configuration (`nomos.json`):** If a `version` key exists in the project configuration, all files in that project default to that version.
3. **Global Configuration (`config.json`):** If a `version` key exists in the global configuration, it acts as the default fallback.
4. **Extension Inference:** If no version is explicitly configured or declared:
   - File extension `.md` is parsed using the **v0** rules.
   - File extension `.nomos` is parsed using the **latest binary version** (defaults to v1).

### Automatic Version Update

Upgrading files to newer spec versions is handled via the `$ nomos update` subcommand.

For example, to migrate a project from v0 to v1, running `nomos update` will:
1. Scan the project for files matching the current version (e.g., `.md` v0 files).
2. Parse them using the v0 rules.
3. Rewrite the parsed task ASTs to `.nomos` files using the v1 rules (converting alphabetical priorities like `(A)` to numeric `(1)`, stripping redundant `::` delimiters, etc.).
4. Update or create the `version` field in `nomos.json` with a value of `1`.
5. Remove the original v0 `.md` files.

ACK
Should solve the versioning problem for good.

## Status and Priority

In v0 both status and priority are defined as a single character.
This can lead to some confusion while reading the file, and editing it.

Using A-Z for priority was inherited from todo.txt; and 26 different priorities is a bit too much.

Move prio to be a number, 0-9 OR 1-9.
Still single char, but more logical and easier to separate intuitively from status.

0 is a special case; Should it mean most important? 1 would be the best fit for that.
Moving it to least important just does not make sense logically (9 < 0 ??).
Removing it as a valid prio would fly in the face of developer expectations though.

Maybe best to define 0 as a valid, but discouraged, `Extreme` prio level.

### Priority Number System

> Quick note: Using 0 as extreme and 5 as the middle priority level.

| Number | Priority Level | Old Priority Chars |
| --- | --- | --- |
| 0 | Extremely Important | - |
| - | - | - |
| 1 | Most Important | A, B, C |
| 2 | Especially Important | D, E, F |
| 3 | Highly Important | G, H, I |
| 4 | Very Important | J, K, L |
| 5 | Important | M, N |
| 6 | Somewhat Important | O, P, Q |
| 7 | Slightly Important | R, S, T |
| 8 | Less Important | U, V, W |
| 9 | Least Important | X, Y, Z |

ACK
No brainer. The number system is much more logical and easier to separate intuitively from status. 0 as `extreme` is a good middle ground.
